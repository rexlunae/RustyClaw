# RustyClaw v0.4.0 Roadmap

## Overview

Four major features:
1. **Desktop client** using dioxus-bulma (GUI alternative to TUI)
2. **SSH transport** for gateway connections (key-based pairing)
3. **Pairing flow** for both TUI and desktop clients
4. **Agent responsiveness** — background long-running tasks, stay responsive

---

## 0. Agent Responsiveness (Core Requirement)

RustyClaw agents must remain responsive during long-running operations. This means:

### Problem
Currently, when an agent is processing a request (e.g., building code, running tests, large API calls), it blocks and cannot respond to new messages until complete.

### Solution: Background Task System

```rust
// New module: rustyclaw-core/src/tasks/
pub struct TaskManager {
    running_tasks: HashMap<TaskId, TaskHandle>,
    completed_tasks: VecDeque<TaskResult>,
}

pub struct Task {
    id: TaskId,
    description: String,
    status: TaskStatus,  // Pending, Running, Complete, Failed
    started_at: Instant,
    progress: Option<String>,
}
```

### Implementation

1. **Task spawning** — Long-running operations spawn as background tasks
   ```rust
   let task_id = task_manager.spawn("Building rustyclaw", async {
       exec("cargo build --release").await
   });
   ```

2. **Immediate response** — Agent responds immediately with task acknowledgment
   ```
   "Started build task (id: abc123). I'll let you know when it's done."
   ```

3. **Status checking** — New tool or command to check task status
   ```
   /tasks          # List running tasks
   /task abc123    # Check specific task
   ```

4. **Completion notification** — Agent proactively notifies when tasks complete
   ```
   "Build task abc123 completed successfully (took 3m 42s)."
   ```

5. **Interrupt handling** — User can send new messages while tasks run
   - Gateway maintains message queue per session
   - Agent processes new messages between task status checks
   - High-priority messages (e.g., "stop", "cancel") handled immediately

### Changes Required

- **Gateway**: Add task manager, background task spawning
- **Protocol**: Add task-related message types (TaskSpawned, TaskStatus, TaskComplete)
- **Agent tools**: `spawn_task`, `check_task`, `cancel_task`
- **Messenger loop**: Non-blocking message processing, interleaved with task checks
- **TUI/Desktop**: Task status panel, progress indicators

### Priority Tasks to Background
- `cargo build` / `cargo test`
- Long file operations
- Web searches / API calls that may hang
- Sub-agent spawns
- Any exec with expected runtime > 10 seconds

## 2. SSH Gateway Transport

### Goals
- Secure remote access without exposing WebSocket ports
- Key-based authentication (no passwords)
- Works with existing SSH infrastructure OR standalone

### Two Modes

#### A. Standalone SSH Server
Gateway runs its own SSH server on a dedicated port.

```toml
[gateway.ssh]
enabled = true
bind = "0.0.0.0:2222"
host_key_path = "~/.rustyclaw/ssh_host_ed25519_key"
authorized_keys_path = "~/.rustyclaw/authorized_clients"
```

**Implementation:**
- Use `russh` crate for SSH server
- Generate host key on first run
- Accept connections, verify client key against authorized_clients
- Create session, bridge SSH channel to gateway protocol

#### B. System SSH Subsystem
Gateway runs as an SSH subsystem under the system's sshd.

```
# /etc/ssh/sshd_config
Subsystem rustyclaw /usr/bin/rustyclaw-gateway --ssh-stdio
```

**Implementation:**
- `--ssh-stdio` flag: read/write protocol on stdin/stdout
- User authenticates via normal SSH (their SSH key)
- Gateway checks if user is authorized in its config
- No separate port needed, uses port 22

### Protocol Over SSH

Same gateway protocol (bincode or JSON), length-prefixed:
```
[4 bytes: length][payload]
```

Or use SSH channel's native framing.

### Crate Changes

```
rustyclaw-core/src/gateway/
├── mod.rs
├── websocket.rs      # Existing WS transport
├── ssh.rs            # NEW: SSH transport
│   ├── server.rs     # Standalone SSH server
│   └── stdio.rs      # Subsystem mode (stdin/stdout)
└── transport.rs      # Trait for WS/SSH abstraction
```

### Dependencies
```toml
# Cargo.toml
russh = { version = "0.50", optional = true }
russh-keys = { version = "0.50", optional = true }

[features]
ssh = ["dep:russh", "dep:russh-keys"]
```

## 3. Pairing Flow

### Client Side (TUI & Desktop)

1. **Generate keypair** (Ed25519)
   - Store in `~/.rustyclaw/client_ed25519_key`
   - Display public key for copy/paste

2. **Pairing UI**
   ```
   ┌─────────────────────────────────────────┐
   │         Pair with Gateway               │
   ├─────────────────────────────────────────┤
   │ Your public key:                        │
   │ ssh-ed25519 AAAA...xyz user@host        │
   │                        [Copy to clipboard]│
   │                                         │
   │ ─── OR scan QR code ───                 │
   │                                         │
   │     ▄▄▄▄▄▄▄ ▄▄▄▄▄ ▄▄▄▄▄▄▄              │
   │     █ ▄▄▄ █ ▀ ▄▀█ █ ▄▄▄ █              │
   │     █ ███ █ █▀▀▄  █ ███ █              │
   │     ▀▀▀▀▀▀▀ ▀ ▀ ▀ ▀▀▀▀▀▀▀              │
   │                                         │
   │ Gateway host: [_________________]       │
   │ Port:         [2222___]                 │
   │                                         │
   │          [Connect]  [Cancel]            │
   └─────────────────────────────────────────┘
   ```

3. **QR code contains:**
   - Client public key
   - Optional: client name/identifier

### Gateway Side

1. **Scan QR / receive public key**
   - Via TUI: `rustyclaw pair --scan` (camera or paste)
   - Via config: manually add to `authorized_clients`

2. **Approve pairing**
   - Show key fingerprint, client name
   - Confirm to add to authorized list

3. **Generate pairing confirmation**
   - Optional: gateway shows its own QR with host:port for easy client config

### Pairing File Format

`~/.rustyclaw/authorized_clients`:
```
# RustyClaw authorized clients
# Format: ssh-ed25519 <key> <name>
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... erica-laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... phone-app
```

## 6. Security Considerations

- **Host key verification**: Client should verify gateway's host key on first connect (TOFU or explicit trust)
- **Key storage**: Private keys should have 600 permissions
- **Subsystem mode**: Inherits SSH's authentication, but gateway should still check authorization
- **No password auth**: Keys only, no fallback

## 5. Embedded Browser (Servo) — Future Feature

**Feature flag:** `browser-servo`

### Rationale
Embedding Servo provides agents with browser automation capabilities without external dependencies:
- No Chrome/Firefox installation required
- No X11/Wayland needed on headless servers
- Sandboxed within the agent process
- Full DOM access via Rust APIs
- Memory-safe browsing (Rust-native)

### Dependencies
```toml
[dependencies]
servo = { version = "0.1", optional = true }  # Check actual crate name/version
# Or embed via libservo if not on crates.io
```

### Integration Points

1. **New tool: `browser_servo`**
   - `navigate(url)` — Load a page
   - `screenshot()` — Capture viewport as PNG
   - `dom_query(selector)` — Query DOM elements
   - `click(selector)` — Click an element
   - `type_text(selector, text)` — Fill input fields
   - `evaluate_js(script)` — Run JavaScript
   - `wait_for(selector | timeout)` — Wait for element/time

2. **Headless rendering**
   - Software rendering via surfman/osmesa
   - No GPU required
   - Configurable viewport size

3. **Session management**
   - Cookie persistence
   - Multiple browser contexts (like incognito)
   - Proxy configuration

### Architecture

```text
┌─────────────────────────────────────────────┐
│              RustyClaw Gateway              │
├─────────────────────────────────────────────┤
│  browser_servo tool                         │
│    ├── ServoEmbedder (manages Servo)        │
│    ├── BrowserContext (cookies, storage)    │
│    └── RenderTarget (headless surface)      │
├─────────────────────────────────────────────┤
│  libservo / servo crate                     │
│    ├── WebView API                          │
│    ├── Constellation (page management)      │
│    └── Layout/Style/Script engines          │
└─────────────────────────────────────────────┘
```

### Build Considerations
- Servo is large (~100MB+ binary impact)
- Long compile times
- Feature-gated to keep default builds small
- May need vendored dependencies

### Prior Art
- `chromiumoxide` — Current approach (external Chrome)
- `headless_chrome` — Similar, requires Chrome
- `webkit2gtk` — GTK-based, needs display server
- `servo` — Fully embeddable, Rust-native
